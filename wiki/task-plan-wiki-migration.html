Phase 1 — Shared System Wiring

Mechanical changes only — no content edits. Apply to every page in the checklist.

Replace inline <style> with shared CSS link

Remove the entire <style>…</style> block from <head>. Replace with:

<link rel="stylesheet" href="/shared/wiki.css" />
Add wiki-nav div

First child of <body>, before .layout:

<div id="wiki-nav"></div>
Wrap content in .layout / .sidebar / .main

Move the page body into <div class="layout"> with a <nav class="sidebar"> and <div class="main">. Add a breadcrumb at the top of .main using the breadcrumb class. The existing page title, content, and nav links move inside .main.

Add wiki-meta div

Immediately after the breadcrumb. All five data-* attributes are required. Use the /projects/kittybarrage/ prefix for data-file-path. Set data-state based on the page content: polished for completed task plans, ongoing for living reference pages, planning for open plans.

<div id="wiki-meta"
  data-file-path="/projects/kittybarrage/PAGENAME.html"
  data-tags="kittybarrage,TAG"
  data-created="2026-05-25"
  data-updated="2026-05-25"
  data-state="polished">
</div>
Add changelog block

After wiki-meta, before the title <h1>. Use "Migrated to shared wiki system" as the first entry with today's date. Add a second entry if there's a meaningful original creation date to preserve.

<details class="changelog">
  <summary>Changelog</summary>
  <ul>
    <li><time>2026-05-25</time> — Migrated to shared wiki system</li>
  </ul>
</details>
Update title row to use h1-row class

Replace any header-row or inline-flex title wrapper with <div class="h1-row">. Keep existing pill badges.

Add script block and initWikiPage()

At the bottom of <body>, before </body>. Include all three scripts in order:

<script src="/shared/wiki.js"></script>
<script src="/shared/nav.js"></script>
<script>
  initWikiPage({ sectionNotes: [] });
</script>

The sectionNotes array gets populated in Phase 2.

Add wiki-copy-bar div

Last element inside .main, after all content:

<div id="wiki-copy-bar"></div>

curl -s -o /dev/null -w "%{http_code}" http://localhost:8090/projects/kittybarrage/PAGENAME.html

Phase 2 — Content Review

After Phase 1, review each page's content structure and add missing wiki system elements.

Add sidebar section links for every <h2>

Each <h2 id="…"> in the page must have a matching <a href="#id"> in the sidebar nav, under an appropriate <div class="s-head">. Scan the full page content — don't guess from memory.

Add section-notes textarea after every <h2> section

Every <h2> section ends with:

<div class="section-notes">
  <label for="notes-SECTIONID">Notes — Section Name</label>

Add an entry for each textarea added in the previous step:

Scan for cross-link opportunities

Every reference to another wiki page by name is a candidate for a link. At minimum, link between pages that are referenced in the same section (e.g., a task plan referencing its architecture page).

Phase 3 — Index Page Redesign

The kittybarrage index.html is a hub page. It currently uses custom .grid / .card classes that exist only in its inline styles. The correct target is the established wiki hub pattern — project-grid / project-card / card-footer — already used by every other hub in the wiki (/life/index.html, root /index.html, etc.). No CSS additions to shared/wiki.css are needed or wanted.

Migration steps

Replace all .grid / .card markup with project-grid / project-card

Each card becomes a <a class="project-card" href="…"> with an <h3> title, <p> description, and <div class="card-footer"> containing status pills. The existing card content (title, description, meta date) maps directly.

Separate active from done task plans

Active / Decided plans stay in a project-grid visible by default. The 7 Done plans move into a <details><summary>Completed (7)</summary> accordion containing a svc-table with columns: Plan | Status | Date. Collapsed by default.

Resolve the Recent Changes log (answer Q2 first)

Pending the answer to Q2, either replace with a svc-table using pill-green / pill-red / pill-blue type badges, or remove the section entirely.

Verify hub structure matches wiki design language

Load http://localhost:8090/projects/kittybarrage/ alongside http://localhost:8090/life/ and confirm visual consistency — same card shape, same type scale, same sidebar structure.

Q-cards below are kept for reference. Decisions are final.

1
Completed task plans — accordion or archived section?

Seven of the 13 task plans are marked Done and have no active planning value. The index hub currently shows them in the same grid as active items. A collapsed <details> accordion keeps the hub focused; a separate "Archived" section keeps them visible but de-emphasised.

2
Recent Changes log — replace with svc-table or remove?

The index has a custom .log / .log-entry component showing ~13 feature/fix entries. It's manually maintained and largely duplicates git log. The question is whether the log serves a purpose that git history doesn't — e.g. as a human-readable "what's new" for someone who doesn't read git output.

3
data-created for pre-wiki pages — migration date or git history?

All 15 kittybarrage pages predate the wiki system and have no known data-created value. The wiki-meta component displays and filters on this date, so it matters for wiki consistency. Options are to use the migration date for all (uniform, simple) or to infer per-page from git history (more accurate, more work).

4
Content freshness — fix stale markers in Phase 2 or flag separately?

Several planning docs describe features that are now fully implemented. Their status markers (pills, data-state) may say "Planning" when the implementation is done. Fixing them during Phase 2 is efficient but risks scope creep; flagging with data-state="needs-work" defers the content review cleanly.

5
architecture.html hub prominence — first card + badge, or evaluate featured layout?

The current index labels architecture.html "Start here." On the redesigned hub, it will be a project-card like all others. The question is whether first position with a "Start Here" pill badge is sufficient, or whether a full-width featured card above the grid would better serve the "read this first" intent — especially for an agent navigating the wiki.
