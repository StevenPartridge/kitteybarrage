Pixel Rooms — Indoor World

Decided May 2026

ToffeCraft Pixel Rooms pack — asset survey complete. TileMap world, furniture scenes, hotspot slots, floor-type behavior, and a house layout for kitties to explore.

Goal & Scope

Give the kitties a real indoor world: rooms built from a pixel-art tileset, furnished with pieces they can navigate around, sit on, sniff, and knock off surfaces. Three interlocking systems:

  • TileMap world — floors and walls painted in Godot's TileMap GUI. Custom data layers on floor tiles encode floor type (stone, wood) which influences kitty behavior.
  • Furniture scenes — all furniture placed as individual scenes with collision, navigation avoidance, and named hotspot slots with finite capacity (sit ×N, lay ×N, sniff, knock).
  • Floor-type behavior — a two-tier lookup: first checks if the kitty is inside a rug/carpet's Area2D (→ RUG), then falls back to the TileMap tile type (STONE or WOOD). ActivityBrain applies bias modifiers at decision time.

Deliverable: a complete multi-room house kitties can explore autonomously. Outdoor assets and the Halloween pack are explicitly future scope.

Technical Context

Sprite crop ownership

Furniture visuals are now editor-owned by default. Set the Sprite2D.texture and crop Sprite2D.region_rect directly in the scene so the editor view and runtime view match. Furniture.gd will not rewrite the sprite region unless apply_definition_sprite_region is enabled.

When apply_definition_sprite_region is enabled, FurnitureDefinition.sprite_w and sprite_h are the stride between atlas cells, not the size of the visible graphic. Variant math computes the region like this:

var col := variant % columns
var row := variant / columns
sprite.region_rect = Rect2(col * sprite_w, row * sprite_h, sprite_w, sprite_h)

So opt-in variant scenes must keep sprite_w equal to sheet_width / columns, even if the visible graphic is smaller than that and surrounded by transparent padding. Getting this wrong can make runtime display differ from the editor. For hand-authored furniture scenes, prefer the scene crop and leave apply_definition_sprite_region disabled.

Observed failure: Bookcase looked correct in the editor but showed two stacked sprites at runtime because bookcase.tres used sprite_h = 64 on a 2×2, 32px-cell sheet. Editor-owned sprite regions prevent this class of mismatch.

A related issue: when a sprite graphic is not centered within its cell, hotspot positions can visually appear off. Prefer moving the Sprite2D, collision shapes, and Hotspots marker nodes in the scene editor so the visual setup remains inspectable.

Furniture StaticBody2D must be on a dedicated physics layer

In Character.gd, _has_line_of_sight() raycasts against WALL_PHYSICS_LAYER = 1. If a furniture scene's StaticBody2D is also on layer 1 (the Godot default), the kitty's ray to a hotspot slot — which is inside the furniture bounds — will hit the furniture's own body and return false. The hotspot is silently skipped every time.

Spec error in the checklist: The blocking-furniture node tree was documented with "StaticBody2D collision layer 1". This is wrong — it must be layer 2 (or any layer ≠ 1). The checklist has been corrected. Any scene already built with layer 1 needs this fixed before hotspots will work.

Approach navigation — implemented, needs bed validation

The slot-claiming system, LOS raycast filter, and approach navigation path are now wired together for SIT and LAY hotspots. Character.apply_decision() sets the hotspot slot as the navigation target and calls begin_walk(); LAY stores an _arrival_state_factory so arrival transitions through LayDownState → LayState. MovementState.tick() calls pop_arrival_state() when the character reaches the target.

Navigation arrival is intentionally checked against the actual target position, not only NavigationAgent2D.is_navigation_finished(), so the agent cannot stop visibly below a hotspot just because Godot considers the path finished. Character._ready() also tightens path_desired_distance and target_desired_distance to 2px.

Observed navigation risk: The current NavigationRegion2D covers the floor space inside the walls, but kitties do not reliably show doorway/narrowing intelligence; they may walk straight into a wall and stop. Treat this as a global navigation diagnosis item, not as proof that bed approach/mount logic is wrong.

Manual testing hook: left-click a visible nearby hotspot with the focused kitty. WorldDirector._input() gives hotspot clicks priority over LookTrack mouse behavior and calls Character.occupy_hotspot_at(), which claims the clicked slot and walks there immediately.

# SIT hotspot
set_target(slot_pos)
begin_walk()        # arrival defaults to SitState

# LAY hotspot
set_target(slot_pos)
_arrival_state_factory = func(): return LayDownState.new(LayState.new())
begin_walk()

Current bed-specific tuning: bed.tres slots were nudged 10px upward, then Bed.tscn's sprite, collision shape, and navigation obstacle were also moved 10px upward so the bed art matches the now-correct kitty/hotspot occupation point. Verify with Furniture.debug_hotspots = true and the new click test before building more scenes.

Two-tier floor type lookup (new insight from survey)

The original plan assumed floor type came entirely from TileMap tile data. The survey revealed that rugs/carpets are standalone sprites — they are not tileset tiles. This requires a two-tier lookup:

# In Character._update_floor_type():
# Tier 1 — check if inside a rug Area2D (registered set)
for rug in _overlapping_rugs:
    if rug.get_overlapping_bodies().has(self):
        current_floor_type = FloorType.RUG
        return

# Tier 2 — read tile custom data from floor layer
var tile_data := floor_layer.get_cell_tile_data(
    floor_layer.local_to_map(floor_layer.to_local(global_position))
)
if tile_data:
    current_floor_type = FloorType[tile_data.get_custom_data("floor_type")]

The simplest implementation: Carpet scenes join a group "rug"; Character holds an Area2D that tracks overlapping rugs via area_entered/area_exited. No need for a registry — just a local set of overlapping rug areas on each Character.

Furniture variant selection

For now, variants are designer-picked by cropping Sprite2D.region_rect in each furniture scene or by enabling editable children on a placed instance and adjusting the child sprite. The root variant field remains available for future bulk workflows, but it only rewrites the crop when apply_definition_sprite_region is enabled.

Navigation: what actually blocks

From the survey: sofas, beds, desks, drawers, bookcases, kitchen counters, oven, fridge, bathtub, toilet, and sink all block navigation. Plants, lamps, and small props do not. Doors in the wall layer must be marked walkable in the TileSet so the navmesh passes through them.

Decisions — Locked In

All five design questions answered. These are the architectural constraints implementation must follow.

1
How many TileMapLayer nodes?
Decided

Question: how to structure the layer hierarchy for floors, walls, and decorations.

Chosen: A — Two layers: Floor + Walls. Floor layer carries all walkable tiles with floor_type custom data. Wall layer holds wall tiles, WalllMaker connectors, and door tiles. Wall decorations (windows, paintings, clocks) placed as Sprite2D scenes — gives per-item control and makes attaching windowsill hotspots trivial.
2
How do furniture color variants get selected?
Decided

Question: runtime random, designer-picked, or resource-driven selection.

Chosen: A — Hand-pick in the editor. Set the visible Sprite2D.region_rect crop for the scene or placed instance. The exported variant field is an opt-in helper only when apply_definition_sprite_region is enabled; visual editor state is the source of truth for normal furniture work.
3
How does hotspot discovery work?
Decided

Question: global registry, proximity scan, or line-of-sight filtered approach.

Chosen: C + raycast filter + Door markers — proximity with line-of-sight. Core principle: kitties only act on furniture they can actually reach without going through a wall. Three parts working together:

1. Interest zone (Area2D, r≈200px): Character detects all nearby Furniture nodes reactively via signals. Area2D naturally goes through walls — this is intentional. A large zone means a kitty can "sense" the bed in the next room.

2. Raycast filter: Before claiming any hotspot, cast a ray from the character to the hotspot's world position against the walls physics layer only. If the ray hits — wall is in the way — skip that candidate. If the ray is clear, the hotspot is reachable.

3. Door markers (EXPLORE waypoint): A small invisible Door.tscn scene is placed at the center of each doorway gap. Current code recognizes Door instances directly and navigates to their global_position; it does not claim a hotspot slot or require a door definition resource. Curiosity weight in KittyPersonality drives how often EXPLORE is chosen.
4
Where does floor-type behavior live?
Decided

Question: should floor-type query live in the Brain, Character, or a global service.

Chosen: A — Cached on Character, polled every 0.5s. Character holds current_floor_type updated by a timer. Rug overlap is maintained reactively via area_entered/exited signals — the 0.5s timer only handles the TileMap tile query. ActivityBrain reads the cached value at decision time. No TileMap coupling in Brain or States.
5
How does furniture block navigation?
Decided

Question: static navmesh bake, dynamic obstacles, or hybrid.

Chosen: B — TileMap physics bake (walls) + NavigationObstacle2D (furniture). Walls bake into the NavigationRegion2D via TileSet physics layer — hard navmesh blocks. Furniture scenes each carry a NavigationObstacle2D for dynamic soft avoidance. Agents route around walls at the macro level, then steer around furniture in real time. Idiomatic Godot 4 approach; sets up correctly for outdoor assets later.

4b — Prove the bed end-to-end before building anything else

The bed is your pilot. All the pieces need to work together for a single piece before you build eleven more. The acceptance test for 4b is the same as Phase 5's test — they converge here.

Definition of "bed done": A kitty sees the bed from across the room → the interest zone picks it up → the brain decides SIT or LAY → the kitty walks across the room to the hotspot slot → arrives and transitions to SitState or LayState → the slot dot turns green (claimed) → on leaving SitState the slot is released and another kitty can claim it. Implemented: approach navigation

A kitty that claims a SIT or LAY hotspot now walks to the slot before occupying it. SIT hotspots call begin_walk() and use the default arrival state. LAY hotspots set an _arrival_state_factory before walking so arrival enters LayDownState → LayState.

MovementState.tick() now uses explicit target-distance arrival and then calls:

return entity.pop_arrival_state()
# returns stored factory result if set, else SitState.new()

This keeps MovementState generic — it doesn't know about SIT vs LAY. The requested action travels with the claimed slot, so a compound SIT_LAY hotspot can satisfy either decision and choose the correct arrival pose. For bare left-click testing, the first token in the group name is the default action. SNIFF and KNOCK actions can use the same pattern when their states are implemented. Manual validation is faster now: left-click a nearby visible hotspot with the focused kitty to claim it and walk there immediately.

Door markers do not claim slots. Once the kitty arrives at the door position, it's crossed into the adjacent room and ActivityBrain fires a new decision cycle. The door is a waypoint, not a seat. This path already works correctly via begin_walk().
