Pixel Rooms — Indoor World

Decided May 2026

ToffeCraft Pixel Rooms pack — asset survey complete. TileMap world, furniture scenes, hotspot slots, floor-type behavior, and a house layout for kitties to explore.

Goal & Scope

Give the kitties a real indoor world: rooms built from a pixel-art tileset, furnished with pieces they can navigate around, sit on, sniff, and knock off surfaces. Three interlocking systems:

  • TileMap world — floors and walls painted in Godot's TileMap GUI. Custom data layers on floor tiles encode floor type (stone, wood) which influences kitty behavior.
  • Furniture scenes — all furniture placed as individual scenes with collision, navigation avoidance, and named hotspot slots with finite capacity (sit ×N, lay ×N, sniff, knock).
  • Floor-type behavior — a two-tier lookup: first checks if the kitty is inside a rug/carpet's Area2D (→ RUG), then falls back to the TileMap tile type (STONE or WOOD). ActivityBrain applies bias modifiers at decision time.

Deliverable: a complete multi-room house kitties can explore autonomously. Outdoor assets and the Halloween pack are explicitly future scope.

Technical Context

Two-tier floor type lookup (new insight from survey)

The original plan assumed floor type came entirely from TileMap tile data. The survey revealed that rugs/carpets are standalone sprites — they are not tileset tiles. This requires a two-tier lookup:

# In Character._update_floor_type():
# Tier 1 — check if inside a rug Area2D (registered set)
for rug in _overlapping_rugs:
    if rug.get_overlapping_bodies().has(self):
        current_floor_type = FloorType.RUG
        return

# Tier 2 — read tile custom data from floor layer
var tile_data := floor_layer.get_cell_tile_data(
    floor_layer.local_to_map(floor_layer.to_local(global_position))
)
if tile_data:
    current_floor_type = FloorType[tile_data.get_custom_data("floor_type")]

The simplest implementation: Carpet scenes join a group "rug"; Character holds an Area2D that tracks overlapping rugs via area_entered/area_exited. No need for a registry — just a local set of overlapping rug areas on each Character.

Furniture variant selection

Each furniture file contains color variants in a grid. Selecting a variant at runtime uses Sprite2D.region_rect — set the atlas rect to the correct grid cell offset. This is the same pattern used for color variants on the kitty sprites.

# Example: Bed.gd — variant 0..3
@export var variant: int = 0

func _ready() -> void:
    var col := variant % 2
    var row := variant / 2
    $Sprite2D.region_rect = Rect2(col * 64, row * 64, 64, 64)

Navigation: what actually blocks

From the survey: sofas, beds, desks, drawers, bookcases, kitchen counters, oven, fridge, bathtub, toilet, and sink all block navigation. Plants, lamps, and small props do not. Doors in the wall layer must be marked walkable in the TileSet so the navmesh passes through them.

Decisions — Locked In

All five design questions answered. These are the architectural constraints implementation must follow.

1
How many TileMapLayer nodes?
Decided

Question: how to structure the layer hierarchy for floors, walls, and decorations.

Chosen: A — Two layers: Floor + Walls. Floor layer carries all walkable tiles with floor_type custom data. Wall layer holds wall tiles, WalllMaker connectors, and door tiles. Wall decorations (windows, paintings, clocks) placed as Sprite2D scenes — gives per-item control and makes attaching windowsill hotspots trivial.
2
How do furniture color variants get selected?
Decided

Question: runtime random, designer-picked, or resource-driven selection.

Chosen: A — Hand-pick per instance in the inspector. Each Furniture scene exports variant: int. Set it in the editor when placing furniture in the room. The room is a designed space — picking the teal sofa for the living room is a creative decision, not a code decision.
3
How does hotspot discovery work?
Decided

Question: global registry, proximity scan, or line-of-sight filtered approach.

Chosen: C + raycast filter + Door markers — proximity with line-of-sight. Core principle: kitties only act on furniture they can actually reach without going through a wall. Three parts working together:

1. Interest zone (Area2D, r≈200px): Character detects all nearby Furniture nodes reactively via signals. Area2D naturally goes through walls — this is intentional. A large zone means a kitty can "sense" the bed in the next room.

2. Raycast filter: Before claiming any hotspot, cast a ray from the character to the hotspot's world position against the walls physics layer only. If the ray hits — wall is in the way — skip that candidate. If the ray is clear, the hotspot is reachable.

3. Door markers (EXPLORE hotspot): A small Door.tscn scene is placed at the center of each doorway gap. It exposes a single EXPLORE-type hotspot slot. When a kitty chooses EXPLORE, it navigates to the door — crossing into the adjacent room. Curiosity weight in KittyPersonality drives how often EXPLORE is chosen.
4
Where does floor-type behavior live?
Decided

Question: should floor-type query live in the Brain, Character, or a global service.

Chosen: A — Cached on Character, polled every 0.5s. Character holds current_floor_type updated by a timer. Rug overlap is maintained reactively via area_entered/exited signals — the 0.5s timer only handles the TileMap tile query. ActivityBrain reads the cached value at decision time. No TileMap coupling in Brain or States.
5
How does furniture block navigation?
Decided

Question: static navmesh bake, dynamic obstacles, or hybrid.

Chosen: B — TileMap physics bake (walls) + NavigationObstacle2D (furniture). Walls bake into the NavigationRegion2D via TileSet physics layer — hard navmesh blocks. Furniture scenes each carry a NavigationObstacle2D for dynamic soft avoidance. Agents route around walls at the macro level, then steer around furniture in real time. Idiomatic Godot 4 approach; sets up correctly for outdoor assets later.

Phase 5 — Hotspot System Complete

No global registry. Each Character carries an interest zone (detects nearby furniture reactively) and a raycast filter (confirms line of sight before claiming). Door markers expose an EXPLORE hotspot that lets kitties deliberately cross into adjacent rooms.

# FurnitureHotspot.gd
class_name FurnitureHotspot
extends Resource

enum ActionType { SIT, LAY, SNIFF, KNOCK, EXPLORE }

@export var action: ActionType
@export var slots: Array[Vector2]  # world offsets from furniture origin
var _claimed: Dictionary = {}      # slot_index → character node
var knocked := false               # one-shot: permanently unavailable after knock

func claim(character) -> int:
    if knocked: return -1
    for i in slots.size():
        if not _claimed.has(i):
            _claimed[i] = character; return i
    return -1

func release(character) -> void:
    for key in _claimed.keys():
        if _claimed[key] == character:
            _claimed.erase(key); return
EXPLORE slots don't stay claimed. Once the kitty arrives at the door position, it's crossed into the adjacent room and ActivityBrain fires a new decision cycle. The EXPLORE slot should be released immediately on arrival — no holding. The door is a waypoint, not a seat.
