Wiki Grooming Prompts Living

Prompts to run with Claude to keep the wiki accurate, current, and useful.

Open a new Claude Code session in the project root, then paste whichever prompt fits.

After a Feature or Refactor

Run these after a coding session that changed how the system works.

Sync architecture.html to current code High value

When: after adding a new state, changing transition logic, or modifying the debug console.

Read wiki/architecture.html and compare it against the current codebase. Identify any sections that are now stale or incorrect — new states, changed transitions, updated ALLOWED table in DebugOverlay.gd, or removed features. List each discrepancy, then update the relevant sections in architecture.html to match the current code. Do not rewrite sections that are still accurate.
Add a new state to architecture.html Targeted

When: after implementing a new State subclass.

I've added a new state: [STATE_NAME]. Read its implementation in scripts/kitty_states/[FileName].gd, then add it to the State Reference section in wiki/architecture.html. Include: state type (rest/transition/movement), constructor params, animation played, what it transitions to, and the enum value. Also update the Transition Graph table with any new paths this state enables, and update the ALLOWED table in the Debug Console section if the debug overlay was updated.
Write a task plan for a new feature High value

When: before starting a significant new feature or refactor.

Write a task plan for: [FEATURE DESCRIPTION]. Save it to wiki/ as an HTML file following the existing wiki design system. The plan should include: goal, affected files, implementation steps, edge cases to handle, and a testing checklist. Link it from wiki/index.html under "Task Plans".

Bug Documentation

Run these when tracking a reproducible issue.

Document a bug High value

When: found a reproducible issue worth tracking.

Document this bug in the wiki. Create wiki/issue-[short-name].html following the wiki design system. Include: description of the observed behavior, steps to reproduce (using debug console if applicable), which states and modules are involved (reference architecture.html terminology), root cause if known, and status (open/investigating/fixed). Link it from wiki/index.html under "Issue Notes".
Close a bug and record the fix Housekeeping

When: a tracked issue has been resolved.

Update wiki/issue-[name].html: mark the status as Fixed, add a "Resolution" section describing what the root cause was and what changed to fix it (file and line references). Also update the Recent Changes log in wiki/index.html with a one-line fix entry dated today.

Periodic Audits

Run occasionally — monthly or after a cluster of changes.

Full wiki staleness audit Periodic

When: after a sprint of work, or when the wiki hasn't been touched in a while.

Audit the entire wiki for staleness. Read every HTML file in wiki/ and cross-reference it against the current codebase. For each page, report: (a) whether it's accurate, (b) specific claims that are now wrong or outdated, (c) things missing that should be documented. Produce a prioritized list of updates needed. Do not make any changes yet — just report.
Update the Recent Changes log Housekeeping

When: after a cluster of fixes or features to record what changed.

Look at the git log since the last entry in wiki/index.html's "Recent Changes" section. Summarize the meaningful changes in 3–5 log entries: what changed, what type it was (fix/feature/refactor), and the date. Update the log in wiki/index.html. Keep entries to one line each.
Debrief after a bug-fixing session Periodic

When: after a session focused on fixing bugs — capture the root causes while they're fresh.

We just fixed these bugs in this session: [LIST BUGS]. For each one, read the relevant code changes and write a short debrief: what the root cause was, which modules were involved, and what the fix was. Add each to wiki/index.html's Recent Changes log. If any of the bugs revealed a pattern worth documenting in architecture.html (a gotcha, an invariant, a constraint), add a note there too.

Wiki Structure

Run these when adding new sections or reorganizing.

Add a new planning document to the index Targeted

When: a new HTML plan was created and needs to be linked from the index.

Add wiki/[filename].html to wiki/index.html. Place it in the appropriate section ([Planning Documents / Task Plans / Issue Notes / Core Reference]). The card should have a title, a one-sentence description, and a date. Use the color class that matches its type (blue for planning, green for reference, gold for task plans, red for issues).
Improve the "How to Describe a Bug" section Periodic

When: a confusing bug report happened and the guide needs a better example.

In wiki/architecture.html, find the "How to Describe a Bug" section. Add a new example based on this real bug we just encountered: [DESCRIBE THE BUG AND THE CONFUSION]. The example should show a bad report (vague), then a good report (precise — state names, trigger type, expected vs actual). Match the existing example format.
Tip: These prompts work best at the start of a fresh Claude Code session when the full codebase context is available. Paste directly into the prompt — no need to add "please" or framing.
