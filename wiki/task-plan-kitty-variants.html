Kitty Color Variants & Markings

Complete May 2026

Give each spawned kitty a unique appearance by swapping sprite sheet variants at runtime — with optional marking overlays and personality ties to color.

Goal & Scope

Right now every kitty is orange. The asset pack ships 52 individual sprite sheet PNGs across 24 color families, plus a separate set of marking overlays (tabby stripes, face markings, white paws, colored ears). The goal is to assign each spawned kitty a distinct appearance so a large group feels like real variety.

There are two separable concerns here:

  • Base color swap — pick one of the 52 sprite sheets at spawn time instead of always using orange_0.png. This is handled via runtime atlas replacement on a duplicated SpriteFrames resource.
  • Markings/overlays — a second composited layer (tabby stripes, white paws, face markings) drawn on top of the base coat. These exist as separate sprite sheets in the asset pack, aligned to the same animation frames. Deferred to a follow-up task.

Color is cosmetic only. Personality is randomized entirely independently — color variant is visual flavor, not a behavioral signal.

Technical Context

How the current sprite is loaded

The Kitty scene (scenes/Kitty.tscn) contains an AnimatedSprite2D node with a SpriteFrames sub-resource baked inline. Each animation frame is an AtlasTexture whose atlas property points to orange_0.png. There are 8-directional variants of each animation state (Sit_North, Walk_East, etc.), resulting in many AtlasTexture objects all sharing the same atlas file.

Sub-resource sharing in Godot 4

When a PackedScene is instantiated in Godot 4, inline sub-resources are not automatically deep-copied per instance. AtlasTexture objects are shared across instances — modifying one kitty's atlas without duplication would corrupt all kitties using the same resource. The fix is to call sprite_frames.duplicate(true) before modifying, forcing a deep copy.

Bug fixed during implementation: The naive atlas-swap loop (replacing .atlas directly on the shared SpriteFrames) modified every kitty that had already been spawned. The duplicate(true) guard in _swap_atlas is the correct solution and must not be removed.

Atlas swap implementation

func _swap_atlas(sprite: AnimatedSprite2D, atlas: Texture2D) -> void:
    sprite.sprite_frames = sprite.sprite_frames.duplicate(true)
    var frames := sprite.sprite_frames
    for anim in frames.get_animation_names():
        for i in frames.get_frame_count(anim):
            var tex := frames.get_frame_texture(anim, i)
            if tex is AtlasTexture:
                (tex as AtlasTexture).atlas = atlas

func apply_color_variant(atlas: Texture2D) -> void:
    _swap_atlas(anim._player, atlas)

Design Decisions

Decisions locked in May 2026. Recorded here for future reference.

1
Sprite swap mechanism — how do we apply a color variant?
Option A

The current scene has one SpriteFrames resource baked in with orange_0.png as the atlas. At spawn time we need to replace the atlas with the chosen variant's PNG.

✓ A — Atlas swap at runtime

After instantiating the Kitty scene, deep-duplicate the SpriteFrames, iterate every AtlasTexture, and replace .atlas. No editor prep, no extra files. Works with the existing scene as-is.

B — Pre-built SpriteFrames per color

Create a separate .tres SpriteFrames for each of the 52 variants in the Godot editor. Clean runtime, significant upfront editor work.

C — Shader palette swap

Keep one sprite sheet; use a shader to remap the orange palette. Won't handle complex patterns like calico or game_boy correctly.

D — Prototype first

Write a proof-of-concept before committing. Superseded — the atlas swap approach was verified immediately during implementation.

Rationale: we already have _swap_atlas, zero editor prep. The duplicate(true) guard handles the Godot 4 sub-resource sharing issue cleanly.
2
Where does color selection live?
Option D

Something has to decide which sprite sheet a kitty gets. The right home for this logic depends on what drives the decision — is it purely random, does the director control the pool, or does the kitty's personality inform it?

A — WorldDirector picks at spawn

Director manages the available color pool and passes a texture path to the kitty after instantiation. Keeps all spawning logic in one place.

B — Kitty picks in _ready()

The kitty randomizes its own color on _ready() from a hard-coded or exported list. Harder to control distribution from the outside.

C — KittyPersonality stores the variant

The personality resource exports a color variant field. Color is set in the inspector, not randomized at runtime.

✓ D — Dedicated ColorVariantPool

A new Resource (ColorVariantPool.gd) manages the available variants, enabled/disabled state, and random selection. Director exports a color_pool reference and calls pick_random() at spawn.

Rationale: color variants depend wholly on character type; ColorVariantPool keeps seams clean as more character types arrive. The pool is also the natural home for the debug overlay's spawn palette UI — the director just queries it.
3
Markings / overlays — do we tackle them in v1?
Option B

The asset pack includes 14 marking overlay sheets (tabby stripes, white paws, face marks, ear colors, tail tips) designed to layer on top of any base coat. They add a lot of visual variety but require additional scene structure and synchronized animation playback.

A — Yes, second sprite layer

Add a second AnimatedSprite2D to the Kitty scene (z_index 1). At spawn, optionally pick a marking overlay and sync it to the same animation name as the base sprite.

✓ B — Not yet, color swap first

Get the base color variant working and visible. Treat markings as a separate follow-up task once the foundation is confirmed stable. Keeps this task's scope clean.

C — Bake markings offline

Use Photoshop/GIMP/Aseprite to composite selected markings into specific color variants. Zero runtime complexity, loses runtime flexibility.

D — Color-family specific only

Some markings only appear on certain color families. Encode as a rule table so markings are contextually appropriate.

Rationale: get color swap stable first; markings as a follow-up. See task-plan-kitty-blemishes.html for the planned marking overlay work.
4
Color → personality relationship
Option C

Should a kitty's coat color say anything about how it behaves? This is purely flavor, but it can make the game feel richer if orange cats are reliably bold while ghost cats are reliably skittish.

A — Hard color presets

Each color family maps to a specific KittyPersonality resource or override set. Players can learn to predict behavior from coat.

B — Soft biases

Color nudges personality weight sliders within their existing random ranges. Behavior is still random, just tilted.

✓ C — Fully independent

Color is cosmetic only. Personality is randomized entirely separately. Simpler. Color variety is purely visual.

D — Personality picks color

Pick personality first, then select a color family that fits the resulting personality. Color is a visual expression of already-decided personality.

Rationale: color is cosmetic, personality is randomized separately. Keeps both systems clean and independently tunable. Color→personality biases can be layered on later without touching the core spawning logic.
5
Color selection pool — how is a variant chosen?
Option D

With 52 variants across 24 families, there are real choices about distribution. Purely uniform random means brown appears 9× more often than ghost. A tiered system or family-based selection can balance this, but adds complexity.

A — Uniform across all 52

Pick randomly from all 52 individual PNGs with equal probability. Brown will dominate; ghost will be rare.

B — One per family (24 families)

Pick randomly from 24 color families, then pick a random variant within the chosen family. Prevents brown from dominating.

C — Rarity tiers

Common/uncommon/rare weights. Special kitties feel earned.

✓ D — Player-configurable palette

An exported color_pool on WorldDirector holds a ColorVariantPool resource. A SPAWN PALETTE section in the debug overlay lets the player enable/disable individual color families. Random pick from the enabled set.

Rationale: spawn palette in debug overlay; random from enabled variants; space-conscious 2-col checkbox grid + All/None buttons. Gives players direct control over visual variety without requiring code changes. All 24 families enabled by default.

Implementation Outline

All steps complete as of May 2026.

Verify sub-resource isolation

Confirmed that inline SpriteFrames sub-resources in Godot 4 are shared across instances. The fix — calling sprite_frames.duplicate(true) before any atlas modification — was implemented in _swap_atlas.

Build ColorVariantPool.gd

New Resource subclass. Defines 24 color families with all 50 variant PNG paths. Each family has an enabled flag (default true). Exposes a pick_random(rng) method that selects a random enabled family then a random variant within it. Returns a loaded Texture2D.

Implement _swap_atlas and apply_color_variant on Character.gd

_swap_atlas(sprite, atlas) deep-duplicates the SpriteFrames then iterates every AtlasTexture frame and replaces .atlas. apply_color_variant(atlas) calls _swap_atlas on anim._player. The duplicate(true) guard ensures no cross-instance resource sharing.

Wire into WorldDirector.gd

Added @export var color_pool: ColorVariantPool and a seeded _rng: RandomNumberGenerator. In initialize_character, the director calls color_pool.pick_random(_rng) and passes the result to character.apply_color_variant() before add_child. This happens before the kitty's first _process tick — no frame flicker.

Add SPAWN PALETTE section to DebugOverlay.gd

A collapsible SPAWN PALETTE section in the debug overlay shows a 2-column checkbox grid of all 24 color families. "All" and "None" buttons toggle the entire palette. Checkboxes write directly to the ColorVariantPool resource on the director — changes take effect immediately for the next spawn.

What Shipped

Delivered — May 2026
  • ColorVariantPool.gd — new Resource; 24 families, 50 variants; pick_random(rng) returns a loaded Texture2D from the enabled set.
  • Character.apply_color_variant + _swap_atlas — iterates every AtlasTexture in the SpriteFrames and replaces the atlas; duplicate(true) guard prevents sub-resource sharing corruption (Godot 4 shares SpriteFrames across instances by default).
  • WorldDirector — color_pool: ColorVariantPool export + seeded _rng; calls apply_color_variant on each spawned character before add_child.
  • DebugOverlay SPAWN PALETTE section — 2-column checkbox grid of all 24 color families; All/None toggle buttons; writes directly to the ColorVariantPool resource for immediate effect on next spawn.

Testing Checklist

  • Spawn 10 kitties — at least 5 distinct coat colors visible, no two looking identical unless they happen to be the same variant
  • Each kitty's color persists through all state changes (Sit → Walk → Run → Lay) without flicker or reset
  • No other kitty's color changes when one kitty is spawned (sub-resource isolation confirmed via duplicate(true))
  • Kitty behavior (FSM, AI, input control) unchanged — color swap is purely visual
  • All existing animations play correctly on the new atlas (frames are correctly cropped — no misaligned pixels)
  • Spawn palette checkboxes in debug overlay update the enabled set; unchecked families never spawn
  • All/None buttons correctly toggle all 24 families in one click
  • Performance: spawning 20+ kitties adds no noticeable frame time vs the prior all-orange baseline
  • No Godot errors about missing textures or invalid resource paths in the Output log
