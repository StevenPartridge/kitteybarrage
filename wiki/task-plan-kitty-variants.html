Kitty Color Variants & Markings

Planning May 2026

Give each spawned kitty a unique appearance by swapping sprite sheet variants at runtime — with optional marking overlays and personality ties to color.

Goal & Scope

Right now every kitty is orange. The asset pack ships 52 individual sprite sheet PNGs across 24 color families, plus a separate set of marking overlays (tabby stripes, face markings, white paws, colored ears). The goal is to assign each spawned kitty a distinct appearance so a large group feels like real variety.

There are two separable concerns here:

  • Base color swap — pick one of the 52 sprite sheets at spawn time instead of always using orange_0.png. This might be trivially simple in Godot, or it might require meaningful prep work depending on how the SpriteFrames resource reacts to runtime atlas replacement.
  • Markings/overlays — a second composited layer (tabby stripes, white paws, face markings) drawn on top of the base coat. These exist as separate sprite sheets in the asset pack, aligned to the same animation frames.

Optionally: color can influence personality — a ghost kitty might be inherently more sprinting-prone; an orange kitty might be bold and sociable. This is flavor, not a hard requirement, but worth deciding upfront so the personality system isn't retrofitted awkwardly.

Technical Context

How the current sprite is loaded

The Kitty scene (scenes/Kitty.tscn) contains an AnimatedSprite2D node with a SpriteFrames sub-resource baked inline. Each animation frame is an AtlasTexture whose atlas property points to orange_0.png. There are 8-directional variants of each animation state (Sit_North, Walk_East, etc.), resulting in many AtlasTexture objects all sharing the same atlas file.

Key Godot 4 fact: sub-resource duplication on scene instantiation

When a PackedScene is instantiated in Godot 4, sub-resources defined inline (not as external .tres files) are duplicated for each instance. This means each spawned kitty gets its own private copy of the SpriteFrames resource — modifying it at runtime only affects that kitty, not others.

Practical implication: We can iterate animated_sprite.sprite_frames, swap the .atlas property on every AtlasTexture frame, and only that kitty's appearance changes. No resource cloning or duplication needed beyond what Godot already does at instantiation.

Potential atlas swap code

func apply_color_variant(sprite: AnimatedSprite2D, atlas: Texture2D) -> void:
    var frames := sprite.sprite_frames
    for anim in frames.get_animation_names():
        for i in frames.get_frame_count(anim):
            var tex := frames.get_frame_texture(anim, i)
            if tex is AtlasTexture:
                (tex as AtlasTexture).atlas = atlas
Needs verification: The assumption above (sub-resources are per-instance) holds for inline sub-resources. If Godot 4 shares AtlasTexture objects across instances in practice, the safe fallback is calling sprite_frames.duplicate(true) before modifying — this forces a deep copy. Worth testing with two kitties before building the full system.

Markings as a second AnimatedSprite2D

A marking overlay could be a second AnimatedSprite2D node (child of Kitty, z_index = 1) with its own SpriteFrames resource built from a marking PNG. Since AnimationController drives the base sprite, the marking sprite would need to mirror the same animation name and be told to play in sync. The marking sprite has a transparent background — it just adds pixels on top.

Design Questions

Answer these before implementation starts. Fill in the radios and notes below, then copy your answers at the bottom.

1
Sprite swap mechanism — how do we apply a color variant?

The current scene has one SpriteFrames resource baked in with orange_0.png as the atlas. At spawn time, we need to replace the atlas with the chosen variant's PNG. The key question is whether we do this at runtime, or pre-build resources ahead of time.

Lean: Option A — if sub-resource isolation holds (and it should based on Godot 4 docs), this is a dozen lines of GDScript and requires zero editor prep. Option D is a quick safety check that confirms before we build.

2
Where does color selection live?

Something has to decide which sprite sheet a kitty gets. The right home for this logic depends on what drives the decision — is it purely random, does the director control the pool, or does the kitty's personality inform it?

Lean: Option A — the director already owns spawning and initialization (initialize_character). Color selection fits there naturally. Option D is the right call later if rarity tiers or player-configurable pools are added.

3
Markings / overlays — do we tackle them in v1?

The asset pack includes 14 marking overlay sheets (tabby stripes, white paws, face marks, ear colors, tail tips) designed to layer on top of any base coat. They add a lot of visual variety but require additional scene structure and synchronized animation playback.

Lean: Option B — start clean with just the color swap so there's a working baseline to test. Markings are a great v2 since the overlay mechanism is straightforward once the base is stable.

4
Color → personality relationship

Should a kitty's coat color say anything about how it behaves? This is purely flavor, but it can make the game feel richer if orange cats are reliably bold while ghost cats are reliably skittish. The question is whether this connection is hard, soft, or nonexistent.

Lean: Option C to start — keeps this task's scope to the visual system. Option B or D can be layered on later with a color-to-personality mapping table without touching the core spawning logic.

5
Color selection pool — how is a variant chosen?

With 52 variants across 24 families, there are real choices about distribution. Purely uniform random means brown appears 9× more often than ghost. A tiered system or family-based selection can balance this, but adds complexity.

Lean: Option B or C — uniform family selection avoids the brown-skew problem and is barely more complex. Rarity tiers (C) add great game feel for the special variants with minimal extra code.

Implementation Outline

Pending design question answers. Steps below are the likely shape — subject to revision based on Q1–Q5 decisions.

Verify sub-resource isolation verify first

Instantiate two Kitties in a test scene. After both are added, modify the SpriteFrames atlas on the first kitty only. Confirm the second kitty is visually unaffected. If they share the resource, add a sprite_frames = sprite_frames.duplicate(true) call in the kitty's _ready().

Build the variant texture registry depends on Q2, Q5

A data structure (Dictionary or exported Array) mapping color families to their available PNG paths. E.g. { "orange": ["res://assets/sprites/cats/orange_0.png", …], "ghost": ["res://assets/sprites/cats/ghost_0.png"] }. Lives in WorldDirector (Option A) or a dedicated autoload (Option D).

Implement apply_color_variant() depends on Q1

For Option A: iterate sprite_frames.get_animation_names(), then for each frame call get_frame_texture(), cast to AtlasTexture, and set .atlas = new_texture. Expose as a method on Kitty (or call from WorldDirector directly on the AnimatedSprite2D node).

Wire into spawning depends on Q2

After character_scene.instantiate() and world_layer.add_child(kitty), select a variant from the registry and call apply_color_variant(). This happens before the kitty's first _process() tick, so no frame flicker.

Markings layer depends on Q3 — skip if B or C

If markings are included: add a second AnimatedSprite2D child node to Kitty.tscn with an initially-empty SpriteFrames. At spawn, optionally select and apply a marking texture using the same atlas-swap approach. The marking sprite plays the same animation name as the base — no extra controller needed.

Color → personality wiring depends on Q4 — skip if C

If personality is tied to color: build a mapping from color family to personality bias values (Option B) or preset resources (Option A). Apply after color is selected and before initialize_character() is called.

Testing Checklist

  • Spawn 10 kitties — at least 5 distinct coat colors visible, no two looking identical unless they happen to be the same variant
  • Each kitty's color persists through all state changes (Sit → Walk → Run → Lay) without flicker or reset
  • No other kitty's color changes when one kitty is spawned (sub-resource isolation confirmed)
  • Kitty behavior (FSM, AI, input control) unchanged — color swap is purely visual
  • All existing animations play correctly on the new atlas (frames are correctly cropped — no misaligned pixels)
  • If markings: overlay renders above the base coat, transparent areas show base coat correctly
  • Performance: spawning 20+ kitties adds no noticeable frame time vs the current all-orange baseline
  • No Godot errors about missing textures or invalid resource paths in the Output log
