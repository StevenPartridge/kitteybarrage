Kitty Marking Overlays

Planning May 2026

Add marking overlays (white paws, tabby stripes, ear and face markings, tail tips) as a second composited sprite layer on top of each kitty's base coat.

Goal & Scope

The color variant system (task-plan-kitty-variants.html) gives each kitty a unique base coat from 24 color families. This task adds the next layer of visual variety: marking overlays composited on top of the base coat at spawn time.

The asset pack ships 14 marking overlay PNGs in assets/sprites/cats/Markings/. Each PNG shares the same frame grid as the base cat sprite sheets, so the same atlas-swap mechanism used for color variants can drive a second AnimatedSprite2D node that renders the marking on top.

There are two core concerns to resolve before implementation starts:

  • Scene structure and sync — where does the marking sprite live (baked into the scene or dynamic), and how does it stay in sync with the base sprite's animation, direction changes, pauses, and speed.
  • Assignment and distribution — how are markings chosen at spawn: probabilistic, always-one, player-configurable, or color-family-influenced.

The goal for v1 is a single marking layer (0 or 1 per kitty). Stacking multiple markings is an explicit non-goal for now — address once the single-layer infrastructure is stable.

Technical Context

How base animation is driven

AnimationController.gd wraps a single _player: AnimatedSprite2D. All animation calls go through it: play_loop, play_once, play_transition, play_pose, hold_frame, change_direction_while_playing, and pause. Nothing outside AnimationController calls _player.play() directly.

To synchronize a marking overlay, something must make the marking sprite play the same animation name, at the same frame, with the same speed_scale and pause state, including direction-change-in-flight. The challenge is that AnimationController tracks in-progress animations and handles partial-progress transitions — a naive "play the same name" call will restart the animation from frame 0.

SpriteFrames for the marking sprite

The simplest approach is to start from the base kitty's SpriteFrames resource: sprite_frames.duplicate(true) gives a deep copy with the same frame regions. Then _swap_atlas replaces the atlas on that duplicate with the marking PNG. The marking sprite has a transparent background — it composites on top as a natural alpha overlay. z_index = 1 on the marking sprite ensures it renders above the base coat.

Known edge cases for sync

  • play_once then play_loop — AnimationController sequences these via a animation_finished signal. A brute-force mirror approach handles this correctly since it copies frame and speed_scale each physics tick.
  • hold_frame — AnimationController calls _player.pause() and _player.frame = N. A mirror approach propagates this automatically. A signal-driven approach misses it.
  • change_direction_while_playing — AnimationController calculates progress and seeks to the same frame in the new animation. A mirror copies frame after the seek, which is correct.
  • pause / unpause — AnimationController sets _player.speed_scale = 0 and restores it. Mirror propagates; signal-driven misses speed_scale changes.

Design Questions

Answer these before implementation starts. Fill in the radios and notes below, then copy your answers at the bottom.

1
How does the marking layer live in the scene?

A marking overlay needs a second AnimatedSprite2D to render on top of the base coat. The question is whether this node always exists in the scene (with visible = false when unused) or is added dynamically only when a marking is assigned.

Lean: A or B. A is cleaner for the scene — always-accessible reference, inspector-friendly, easy to verify in the editor. B avoids an always-present node when most kitties may have no marking. Both are straightforward; the choice mainly affects whether Kitty.tscn changes.

2
How is the marking sprite synchronized with the base animation?

AnimationController drives the base sprite through all animation states: play_loop, play_once, play_transition, hold_frame, change_direction_while_playing, pause. The marking sprite must track all of these — not just the animation name, but also frame, speed_scale, and pause state.

Lean: A is cleanest long-term — sync is structural, not inferred. B is the fastest to implement and handles all edge cases with zero AnimationController risk. C is insufficient given the complexity of AnimationController's state. D is a more general version of A if multiple overlays are anticipated.

3
How is the marking sprite's SpriteFrames built?

The marking sprite needs a SpriteFrames resource that has the same animation names and frame regions as the base cat sprite, but with a different atlas image — the marking PNG instead of the coat PNG. The question is how this resource is constructed.

Lean: A — the _swap_atlas approach already handles deep duplication and atlas swapping. Markings are structurally the same problem as color variants applied to a second sprite node. Reusing the existing mechanism keeps the codebase consistent and requires no new tools.

4
How are markings assigned at spawn?

With 14 marking overlays available, the spawning system needs a strategy for deciding whether a kitty gets a marking and which one. This mirrors the same question that was decided for color variants (player-configurable palette), but markings may warrant a different approach given they're less frequent than coat colors.

Lean: C (consistent with how color palette was handled — keeps the pattern uniform) or A (simpler implementation with one tunable probability slider). D adds flavor but requires a data table coupling colors to markings, which may be premature before seeing how the overlays look in practice.

5
How many marking layers per kitty?

The marking system could support stacking — a kitty could get tabby stripes plus white paws plus a face mark. Each additional layer requires another sprite node and another sync mechanism. This question sets the v1 scope.

Lean: A — get one layer working first. Stacking is a natural follow-up once the infrastructure is stable and the sync approach is proven across all animation edge cases. The scene and sync mechanism for A extends cleanly to B or C without rework.

Implementation Outline

All steps below are blocked pending design question answers. The shape shown assumes Q1=A or B, Q2=A or B, Q3=A, Q4=C, Q5=A — revise if decisions differ.

Build MarkingPool.gd new resource

New Resource subclass mirroring ColorVariantPool. Lists all 14 marking PNG paths with per-marking enable/disable flags. Exposes a pick_random(rng) method that returns a loaded Texture2D from the enabled set (or null if none are enabled or the probability roll fails). Probability scalar configurable as an exported property.

Add marking sprite to scene or create dynamically depends Q1

If Q1=A: add a second AnimatedSprite2D child to Kitty.tscn with z_index=1 and visible=false initially. Reference it in Character.gd via @onready var marking_sprite: AnimatedSprite2D = $MarkingSprite.
If Q1=B: create the node in code at spawn time and call add_child on the kitty before it enters the scene tree.

Implement sync mechanism depends Q2

If Q2=A: add optional _overlay: AnimatedSprite2D to AnimationController. Thread it through play_loop, play_once, play_transition, play_pose, hold_frame, change_direction_while_playing, and pause.
If Q2=B: in Character._physics_process, check if marking_sprite is set and copy anim._player.animation, anim._player.frame, and anim._player.speed_scale to marking_sprite each tick.

Apply marking at spawn using _swap_atlas depends Q3=A

If Q3=A: in Character.apply_marking(atlas), duplicate the base SpriteFrames (anim._player.sprite_frames.duplicate(true)), assign to marking_sprite.sprite_frames, then call _swap_atlas(marking_sprite, atlas). Set marking_sprite.visible = true.

Wire MarkingPool.gd into WorldDirector.gd

Add @export var marking_pool: MarkingPool to WorldDirector. In initialize_character, after apply_color_variant, call marking_pool.pick_random(_rng). If non-null, call character.apply_marking(texture). Order matters: color variant must be applied first so the marking sprite has the correct SpriteFrames to duplicate from.

Add Marking Palette section to DebugOverlay.gd depends Q4=C

Mirror the SPAWN PALETTE section from color variants. A collapsible MARKING PALETTE section in the debug overlay shows a 2-column checkbox grid for all 14 markings. A probability slider controls the chance of any marking spawning. All/None buttons. Checkboxes write directly to the MarkingPool resource on the director.

Testing Checklist

  • Spawn 10 kitties — some have markings visible on top of coat, others are bare coat with no marking node visible
  • Marking stays in sync through all animation states: Walk, Run, Sprint, Sit, Lay, LookAround, and all transitions between them
  • Marking stays in sync when direction changes mid-animation (change_direction_while_playing path)
  • Marking stays in sync through hold_frame states (paused on a specific frame)
  • Marking stays in sync when kitty is paused and unpaused (speed_scale = 0 propagated)
  • Color and marking are independent — any base coat color can appear with any marking
  • Disabling a marking in the debug overlay palette prevents it from spawning on subsequent kitties
  • Unmarked kitties are visually identical to kitties before this feature shipped — no regression
  • No Godot errors about missing textures or invalid resource paths in the Output log
  • Performance: marking sprites at 20+ kitties add no noticeable frame cost compared to the unmarked baseline
