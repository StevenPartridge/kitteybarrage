State Topology Refactor

Task Plan May 2026

Move state transition topology out of Character._plan_transition and into per-character data tables, so adding an activity or a new character type never requires editing the base class.

The Problem

Character._plan_transition (and its helper _plan_movement) is a 50-line match statement that encodes every state-to-state path in the game. It knows that going from LAY to WALK requires a situp chain, that LOOK_AROUND can only be entered from SIT, that movement is blocked if a transition animation is already playing. All of this knowledge is baked into the base Character class.

This creates two problems:

Wrong owner

The chains — situp before standup before walk — are kitty-specific anatomy. A snake or wheelchair character has different rules. They all inherit a match statement written for one body type.

Growing match

Every new activity means editing Character.gd, Global.StateName, and ActivityBrain. The topology is implicit in code, not declared as data.

The deletion test confirms the friction: delete _plan_transition and its complexity reappears in every caller. Nothing in the state definitions tells you how they connect — the topology is nowhere except inside the match.

Design

The topology is expressed as two tables, built once at _ready and cached. Character provides base implementations; subclasses extend them.

Posture groups

States are grouped by the character's physical posture — which determines what intermediate steps are needed before a transition can complete. The mapping from StateName to posture group is per-character data.

-- Kitty posture groups -- STANDING ← WALK, RUN, SPRINT SITTING ← SIT, LOOK_AROUND LYING ← LAY, LAY_DOWN TRANSITIONING ← STANDUP, SITUP (all outbound transitions blocked)

A base Character — no special animations — maps all states to STANDING. Only Kitty needs SITTING, LYING, and TRANSITIONING. A future snake character could define its own groups without touching the base.

Transition table

The table maps posture group → activity → Callable. Each entry is a zero-argument callable that returns the full state chain for that transition. A null value means the transition is explicitly blocked. A missing entry is an error.

-- Kitty SITTING row -- SITTING → WALK → func(): return StandUpState.new(MovementState.walk()) SITTING → RUN → func(): return StandUpState.new(MovementState.run()) SITTING → LAY → func(): return LayDownState.new(LayState.new()) SITTING → SIT → null (same state, handled by from==to guard) -- Kitty LYING row -- LYING → WALK → func(): return SitUpState.new(StandUpState.new(MovementState.walk())) LYING → SIT → func(): return SitUpState.new(SitState.new(true, true)) LYING → LOOK_AROUND → null (blocked) -- Kitty TRANSITIONING row -- TRANSITIONING → *all* → null (blocked — don't interrupt a transition)

Lookup

The new _plan_transition is three lines: look up the posture group for from, look up the factory for to, call it or return null.

Design Decisions

Key granularity — fine-grained [from, to] or coarse-grained [posture, to]?
Coarse-grained: posture group → activity.
The from-state only matters insofar as it implies a posture. SIT and LOOK_AROUND require the same standup chain regardless of which one you're in. Grouping collapses ~15 entries to 5–6 and makes the table declarative rather than exhaustive.
Table ownership — base class only, or base + subclass extension?
Character defines a base table; subclasses extend it by calling super and merging.
The base case is "direct transitions everywhere" — correct for any character with no intermediate animations (snake, wheelchair). Kitty overrides SITTING and LYING rows to insert chains. New character types inherit the base and only override what differs.
Blocked transitions — absent from table, or explicit null value?
Explicit null in the table.
Absence is ambiguous — it could mean "not yet implemented" or "intentionally blocked." Explicit null makes blocking a deliberate declaration. A missing entry triggers push_error, which surfaces gaps during development rather than silently doing nothing.
PostureGroup location — Character inner enum, or Global?
Inner enum on Character: enum PostureGroup { STANDING, SITTING, LYING, TRANSITIONING }.
GDScript enums cannot be extended in subclasses, but they can be used from subclasses — Kitty accesses PostureGroup.SITTING directly (no prefix needed when extending). Character is the right owner: posture vocabulary belongs to character anatomy, not the engine. A future snake that needs COILED adds one entry to Character's enum. Global stays free of character-type-specific concepts.

Interface

Character.gd

# Built once in _ready(), cached for the lifetime of the character.
var _posture_groups: Dictionary   # StateName → PostureGroup
var _transition_table: Dictionary  # PostureGroup → { StateName → Callable/null }

# Subclasses override these to extend the base tables.
func _build_posture_groups() -> Dictionary:
    return {
        Global.StateName.WALK:   PostureGroup.STANDING,
        Global.StateName.RUN:    PostureGroup.STANDING,
        Global.StateName.SPRINT: PostureGroup.STANDING,
        # STANDUP and SITUP: TRANSITIONING — all outbound blocked
        Global.StateName.STANDUP: PostureGroup.TRANSITIONING,
        Global.StateName.SITUP:   PostureGroup.TRANSITIONING,
    }

func _build_transition_table() -> Dictionary:
    return {
        PostureGroup.STANDING: {
            Global.StateName.WALK:        func(): return MovementState.walk(),
            Global.StateName.RUN:         func(): return MovementState.run(),
            Global.StateName.SPRINT:      func(): return MovementState.sprint(),
            Global.StateName.SIT:         func(): return SitState.new(),
            Global.StateName.LAY:         null,  # can't go directly from standing to lay
            Global.StateName.LOOK_AROUND: null,  # must be sitting first
        },
        PostureGroup.TRANSITIONING: {
            Global.StateName.WALK:        null,
            Global.StateName.RUN:         null,
            Global.StateName.SPRINT:      null,
            Global.StateName.SIT:         null,
            Global.StateName.LAY:         null,
            Global.StateName.LOOK_AROUND: null,
        },
    }

# Unchanged call-site — callers never touch the table directly.
func _plan_transition(from: Global.StateName, to: Global.StateName) -> State:
    if from == to: return null
    var posture = _posture_groups.get(from, PostureGroup.STANDING)
    var row: Dictionary = _transition_table.get(posture, {})
    if not row.has(to):
        push_error("_plan_transition: no entry for " + str(from) + " → " + str(to))
        return null
    var factory: Variant = row[to]
    return factory.call() if factory != null else null

Kitty.gd

func _build_posture_groups() -> Dictionary:
    var groups = super._build_posture_groups()
    groups[Global.StateName.SIT]          = PostureGroup.SITTING
    groups[Global.StateName.LOOK_AROUND]   = PostureGroup.SITTING
    groups[Global.StateName.LAY]           = PostureGroup.LYING
    groups[Global.StateName.LAY_DOWN]      = PostureGroup.LYING
    return groups

func _build_transition_table() -> Dictionary:
    var table = super._build_transition_table()
    table[PostureGroup.SITTING] = {
        Global.StateName.WALK:        func(): return StandUpState.new(MovementState.walk()),
        Global.StateName.RUN:         func(): return StandUpState.new(MovementState.run()),
        Global.StateName.SPRINT:      func(): return StandUpState.new(MovementState.sprint()),
        Global.StateName.SIT:         func(): return SitState.new(true, true),
        Global.StateName.LAY:         func(): return LayDownState.new(LayState.new()),
        Global.StateName.LOOK_AROUND: func(): return LookAroundState.new(),
    }
    table[PostureGroup.LYING] = {
        Global.StateName.WALK:        func(): return SitUpState.new(StandUpState.new(MovementState.walk())),
        Global.StateName.RUN:         func(): return SitUpState.new(StandUpState.new(MovementState.run())),
        Global.StateName.SPRINT:      func(): return SitUpState.new(StandUpState.new(MovementState.sprint())),
        Global.StateName.SIT:         func(): return SitUpState.new(SitState.new(true, true)),
        Global.StateName.LAY:         null,  # already lying
        Global.StateName.LOOK_AROUND: null,  # must sit up first, then stand
    }
    return table
Note: _plan_movement is deleted entirely. Its logic is now fully captured in the SITTING and LYING table rows above — no separate helper needed.

Implementation Steps

Add PostureGroup to Global

Add enum PostureGroup { STANDING, SITTING, LYING, TRANSITIONING } to Global.gd. No other changes to Global — the enum will grow as new character types are added.

Rewrite Character.gd

Add _posture_groups and _transition_table as instance variables. Add _build_posture_groups() and _build_transition_table() with base implementations. Replace _plan_transition with the three-line table lookup. Delete _plan_movement. Call both build methods in _ready().

Extend tables in Kitty.gd

Override _build_posture_groups() and _build_transition_table(). Call super, merge in SITTING and LYING entries, return. All kitty-specific topology moves here and out of Character.

Verify in Godot

Run the scene. Exercise all transitions manually: walk from sit, walk from lay, look around, lay down from sit, stand up mid-transition (should be blocked). Check that push_error never fires for legitimate transitions.
