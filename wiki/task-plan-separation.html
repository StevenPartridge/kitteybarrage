Character Separation & Avoidance

Task Plan May 2026

Characters steer around each other while walking. Characters at rest don't stack on top of each other. Applies to kitties and future character types.

Concepts & Vocabulary

Game Dev Aside — Steering Behaviors

In 1987, Craig Reynolds published a paper describing how autonomous agents could move in natural-looking ways using simple mathematical rules called steering behaviors. They're still the foundation of how most games handle character movement. The three you care about here:

Separation — each agent pushes away from neighbors within a personal-space radius. The force grows stronger as the distance shrinks. This is what prevents stacking at rest and causes agents to drift apart naturally when they stop near each other.

Seek / Arrive — a force that pulls the agent toward a target. At-distance it's full strength; as it gets close it tapers off so the agent slows to a stop rather than overshooting. Your current MovementState implements a version of this.

Avoidance — predicting that another agent will occupy a position and steering around it before the collision happens. This is what "going around" a blocking kitty requires, as opposed to just being pushed aside reactively.

Combining Seek + Separation gives you agents that move toward goals while naturally flowing around each other — which is exactly what this feature calls for.

Game Dev Aside — RVO and Why We're Not Using It

The academically correct algorithm for multi-agent avoidance is RVO2 (Reciprocal Velocity Obstacles). Godot 4 has it built in via NavigationAgent2D with avoidance_enabled = true. It produces near-perfect avoidance behavior.

We're not using it here because it requires moving velocity application into an asynchronous signal callback (velocity_computed), which would significantly restructure MovementState's synchronous tick() model.

For a desktop pet with 5–10 characters on an open screen, separation forces produce visually equivalent results with a fraction of the complexity. If avoidance ever needs to be more precise (tight corridors, complex layouts), RVO is the upgrade path.

Two Distinct Problems

The feature request describes two different situations that need different solutions:

Problem 1 — En-Route

A walking kitty should steer around a kitty that's in its path. The blocker may be moving or stopped.

Problem 2 — At Rest

Two kitties that stop near each other should not sit directly on top of one another. They should settle slightly apart.

Both are solved by the same underlying mechanism — a separation force that pushes kitties apart when they're within a personal-space radius — but applied at different moments and with different strengths.

Approach

Separation Force

Every frame, for each pair of characters closer than a separation_radius, compute a repulsion vector pointing from one toward the other. The force scales with proximity — full strength when nearly overlapping, zero at the edge of the radius. Add this force to the character's velocity when moving, and apply it as a gentle position nudge when at rest.

kitty_a ←──── repulsion ────→ kitty_b distance = 30px, radius = 64px strength = (1.0 - 30/64) = 0.53 → force = 0.53 × separation_strength

When a walking kitty passes close to a resting one, the separation force steers the walker sideways. The resting kitty gets a tiny counter-force that's small enough to barely notice. Together they look like natural personal-space awareness.

Why not NavigationAgent2D?

Godot's built-in RVO avoidance is async — velocity is returned via a signal, not inline with tick(). Retrofitting that into the state machine would mean restructuring MovementState significantly. Separation forces slot directly into the existing synchronous model.

Architecture

The key question: who computes the separation force, and how does it reach the state?

WorldDirector is the only place that sees all characters simultaneously. States only know about their own entity. So the computation lives in WorldDirector, and the result is placed on Character as a property that MovementState reads.

WorldDirector._physics_process() 1. _compute_separation_forces() ← iterates all pairs, writes kitty.separation_force 2. per-kitty movement / brain tick ← unchanged MovementState.tick() desired_velocity = direction × speed entity.velocity = desired_velocity + entity.separation_force move_and_slide() WorldDirector._physics_process() (rest case) 3. _apply_rest_separation(delta) ← position nudge for stopped kitties

Character gains one new property: separation_force: Vector2. It is written by WorldDirector and read by MovementState. Nothing else changes in the state machine.

Implementation Steps

Add separation_force to Character.gd Already done

Added during the Character generalization refactor. Already present on line 13 of Character.gd:

var separation_force: Vector2 = Vector2.ZERO

Plain var, not exported — per-frame computed value, not a tunable parameter. No action needed.

Add separation constants to WorldDirector.gd

Two tunable values at the top of the file:

const SEPARATION_RADIUS:   float = 64.0   # pixels — personal-space bubble
const SEPARATION_STRENGTH: float = 120.0  # max push speed in px/s when nearly overlapping
const REST_PUSH_STRENGTH:   float = 20.0   # gentler nudge for stopped kitties
Add _compute_separation_forces() to WorldDirector.gd

Call this at the start of _physics_process, before the per-kitty movement pass. It clears last frame's forces and recomputes them:

func _compute_separation_forces() -> void:
    for character in characters:
        character.separation_force = Vector2.ZERO

    for i in characters.size():
        for j in range(i + 1, characters.size()):
            var a: Character = characters[i]
            var b: Character = characters[j]
            var delta: Vector2 = a.position - b.position
            var dist: float   = delta.length()
            if dist >= SEPARATION_RADIUS or dist < 0.01:
                continue
            var strength: float  = (1.0 - dist / SEPARATION_RADIUS) * SEPARATION_STRENGTH
            var push: Vector2    = delta.normalized() * strength
            a.separation_force += push
            b.separation_force -= push

O(N²) over pairs — optimal for small N. With 10 kitties that's 45 pair checks per frame, negligible.

Blend separation_force into MovementState.gd

In tick(), add the separation force to the velocity before moving. The force is already in px/s units, matching velocity:

# Before
entity.velocity = direction * entity.speed * _speed_fn.call(entity.personality)

# After
entity.velocity = direction * entity.speed * _speed_fn.call(entity.personality) \
    + entity.separation_force

When the kitty has no target and is pausing, entity.velocity is set to Vector2.ZERO in the else branch. Leave that as-is — the rest nudge (next step) handles the stopped case separately.

Add _apply_rest_separation() to WorldDirector.gd

For kitties that are not currently in a movement state, apply the separation force as a direct position offset. This handles the "stopped too close" case without needing the state machine involved:

func _apply_rest_separation(delta: float) -> void:
    for character in characters:
        var state: Global.StateName = character.state_machine.current_state_name()
        if state in [Global.StateName.WALK, Global.StateName.RUN, Global.StateName.SPRINT]:
            continue  # moving characters handle it via separation_force in velocity
        if character.separation_force == Vector2.ZERO:
            continue
        var nudge: Vector2 = character.separation_force.normalized() \
            * minf(character.separation_force.length(), REST_PUSH_STRENGTH) * delta
        character.position += nudge

The minf clamp keeps the nudge gentle — stopped kitties drift apart slowly rather than snapping. delta makes it frame-rate independent.

Wire it together in _physics_process

Add both calls to the top of the existing loop in WorldDirector.gd:

func _physics_process(delta: float) -> void:
    _compute_separation_forces()      # ← new, first
    for character in characters:
        # ... existing player/AI movement logic unchanged ...
        pass
    _apply_rest_separation(delta)     # ← new, after movement pass

Tuning

ConstantEffect of increasingEffect of decreasingStarting value
SEPARATION_RADIUS Kitties start steering earlier, more personal space Only separate when nearly overlapping 64.0 px
SEPARATION_STRENGTH Harder, more urgent steering deflection while moving Softer, barely noticeable nudge 120.0 px/s
REST_PUSH_STRENGTH Stopped kitties drift apart faster Very slow drift — subtle but present 20.0 px/s
Tuning tip: Set SEPARATION_RADIUS to roughly the visual width of a kitty sprite. SEPARATION_STRENGTH should be less than walk speed (which appears to be ~150 px/s) so that a walking kitty can still make progress toward its target even when being pushed sideways.
Edge case — exact overlap: If two kitties spawn at the exact same position, delta.length() < 0.01 skips the pair to avoid a zero-vector normalize crash. They'll separate naturally once they start moving. To be safe, offset spawn position slightly if a kitty already exists there (future improvement).

Future Character Types

The separation pass in WorldDirector currently operates on Array[Character]. When other character types arrive, there are two paths:

  • Minimal: Each character type's director runs its own separation pass over its own array. Kitties and dogs don't avoid each other, but that may be fine.
  • Unified: Extract a _compute_separation_forces(characters: Array) free function (or a SeparationSystem node under WorldLayer) that operates on any array of Node2D objects that have a separation_force property. All character types register with it. Cross-species avoidance works automatically.

The unified approach is the right end state. The minimal approach is fine until a second character type actually exists. Don't build the abstraction before it's needed.

Testing Checklist

  • Spawn 2 kitties close together — they should drift apart slowly while at rest
  • Spawn 2 kitties on top of each other — they should separate without teleporting or snapping
  • Send a walking kitty toward a sitting kitty — walker steers around, doesn't walk through
  • Send two kitties toward each other — both deflect, neither stops dead
  • Walking kitty with a distant target passing near many stopped kitties — reaches destination despite deflections
  • 3+ kitties in a cluster — all gradually spread out, no one gets stuck in a corner of forces
  • Single kitty alone on screen — no separation force, movement completely unchanged
  • Kitty in LOOK_AROUND near another kitty — rest separation still nudges them apart
  • No jitter or oscillation when two kitties are just outside separation radius
  • Performance: 8+ kitties on screen, no noticeable frame drop
