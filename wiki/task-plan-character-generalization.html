Character Generalization

Done May 2026

Extract a Character base class from Kitty so the FSM, states, brain, and director work with any character type — not just kitties.

Goal & Scope

Currently State.entity is typed as Kitty and FiniteStateMachine casts to Kitty. Adding a Dog, Bird, or any other character means either duplicating the entire state system or hitting a type wall immediately.

This refactor introduces a Character base class. Kitty becomes a thin subclass. States, the FSM, brains, and the director all target Character. After this lands, adding a new character type is: create a scene, extend Character, done.

Before

  • State.entity: Kitty
  • FSM casts get_parent() as Kitty
  • KittyDirector: Array[Kitty]
  • ActivityBrain takes KittyPersonality
  • All logic lives in Kitty.gd

After

  • State.entity: Character
  • FSM casts get_parent() as Character
  • Director: Array[Character]
  • ActivityBrain takes CharacterPersonality
  • Kitty.gd is a thin subclass or empty

This refactor does not add new features. Behaviour is identical after it lands. It's purely structural.

Design Decisions

Decisions locked in May 2026. Recorded here for future reference.

1
Director architecture — one director or one per character type?
Decided

The current KittyDirector manages player focus, AI brains, spawning, and input — all for kitties. When Dogs exist, should one WorldDirector manage all characters together, or should a DogDirector appear alongside KittyDirector with a thin coordinator above them?

Chosen: Option A — One WorldDirector. Rename KittyDirector → WorldDirector. It manages Array[Character]. Kitty-specific spawning becomes one method among many. Focus cycling, brains, and player control all live in one place. Rationale: The current director is already doing "manage all kitties as a group" — generalising to "manage all characters as a group" is a natural extension. Per-type directors add coordination complexity not yet needed. Can always split later if one type's logic becomes unwieldy.
2
Animation names — enforced convention or character-provided?
Decided

States currently hardcode animation names: SitState plays "Sit", MovementState plays "Walk"/"Run"/"Sprint", LayDownState plays "Lay", LookAroundState plays "LookAround". For a Dog to reuse SitState, does it need an animation named exactly "Sit"? Or should each character declare its own animation names?

Chosen: Option A — Enforced convention. All character types must provide animations named "Sit", "Walk", "Run", "Sprint", "Lay", "LookAround". States stay as-is. Zero state code changes. Rationale: Simple convention holds for most character types. Option B is available as a virtual method override if a character genuinely needs different names — don't pay for the flexibility until it's needed.
Note: A snake would use WALK/SIT/LOOK_AROUND but not LAY/SPRINT. Unused activities are handled by setting preference weight = 0 in the personality — they're never chosen. The animation named "Walk" would show slithering. If the semantic mismatch becomes a problem, Option B can be retrofitted as a virtual method without touching shared states.
3
Personality — shared base or per-character resource?
Decided

KittyPersonality has two kinds of fields: generic ones shared by any character (preference_walk, duration_sit, rest_threshold, speed multipliers) and kitty-specific ones (look_around_direction, look_around_speed, etc.). ActivityBrain reads the generic fields. The look-around fields are only used by LookAroundState.

Chosen: Option A — Thin base + subclasses. CharacterPersonality holds the generic fields. KittyPersonality extends CharacterPersonality adds look-around and any other kitty specifics. ActivityBrain takes CharacterPersonality. Rationale: The base/subclass split keeps future character personalities clean — a Dog's inspector shouldn't show look-around sliders. GDScript Resources support this natively. Explicit types over stringly-typed dictionaries.

New Character.gd — Interface Contract

This is the public interface all future character types inherit. Everything here becomes the contract states and the director can depend on.

extends CharacterBody2D
class_name Character

# Core components
var state_machine: FiniteStateMachine
var anim:          AnimationController

# Inspector-configurable
@export var personality:       CharacterPersonality
@export var initial_activity:  Global.StateName = Global.StateName.SIT
@export var speed:             float = 150.0

# Runtime state
var facing_direction:  Global.Direction = Global.Direction.SOUTH
var navigation_target: NavigationTarget = null
var separation_force:  Vector2          = Vector2.ZERO

# Public interface (called by director and states)
func begin_walk()  -> void: ...
func begin_run()   -> void: ...
func begin_sprint() -> void: ...
func apply_decision(decision: ActivityDecision) -> void: ...
func change_state(state: State)                 -> void: ...
func set_target(target: NavigationTarget)        -> void: ...
func clear_target()                                -> void: ...
func set_highlight(enable: bool)                  -> void: ...
func update_direction(vec: Vector2)               -> void: ...
set_highlight implementation note: Currently Kitty.set_highlight accesses $AnimatedSprite2D directly. On Character, this becomes anim._player.modulate = ... — but _player is private. The cleanest fix: add a set_modulate(color: Color) method to AnimationController. One extra line there, no scene changes needed.

Implementation Steps

Steps 1–5 have no dependencies on the design questions. Steps 6–7 assume recommendations are accepted — flag if not.

Add set_modulate() to AnimationController.gd

One new method so Character.set_highlight doesn't need to access _player directly:

func set_modulate(color: Color) -> void:
    if _player != null:
        _player.modulate = color
Create CharacterPersonality.gd

Extract all generic fields from KittyPersonality: preference weights, durations, rest_threshold, walk_target_chance, run/sprint multipliers. Same @export_group layout. Extends Resource.

Do not remove fields from KittyPersonality yet — strip it in step 4 after Character.gd is working.

Create Character.gd

Copy everything from Kitty.gd into a new file. Change the first two lines to extends CharacterBody2D / class_name Character. Change personality: KittyPersonality to personality: CharacterPersonality. Replace $AnimatedSprite2D.modulate in set_highlight with anim.set_modulate(...). Remove grid_size (unused).

At this stage both Character.gd and Kitty.gd exist as-is — nothing is broken yet.

Strip Kitty.gd and KittyPersonality.gd

Kitty.gd becomes:

extends Character
class_name Kitty

KittyPersonality.gd becomes extends CharacterPersonality / class_name KittyPersonality and keeps only the look-around fields (look_around_direction, look_around_speed, look_around_pause_right, look_around_pause_left, look_around_pause_center, look_around_repetitions).

Verify in Godot that the Kitty scene still loads and the inspector shows the correct personality fields (inherited + extended).

Update State.gd and FiniteStateMachine.gd

State.gd line 5: var entity: Kitty → var entity: Character

FiniteStateMachine.gd line 23: get_parent() as Kitty → get_parent() as Character. Update the error message on line 25 to say "FiniteStateMachine must be a child of Character."

All state files in kitty_states/ inherit the updated type automatically — no edits needed there.

Update ActivityBrain.gd

Change _init and refresh_personality signatures: personality: KittyPersonality → personality: CharacterPersonality. No logic changes — all the fields ActivityBrain reads (preference_walk, duration_sit, etc.) are now on CharacterPersonality.

Update KittyDirector.gd depends on Q1

Change type annotations throughout: Array[Kitty] → Array[Character], controlled_kitty: Kitty → controlled_character: Character, update initialize_kitty parameter type.

If Q1 = rename to WorldDirector: rename the file and class name, update Main.tscn's script reference, update DebugOverlay's find_child("KittyDirector"... call.

Logic — the brain loop, focus cycling, spawning — is completely unchanged.

Update DebugOverlay.gd

Update type annotations for _director, update controlled_kitty references to controlled_character, update the find_child name if director was renamed. The UI, button handlers, and personality panel are fully reusable for any character — no logic changes.

What Doesn't Change

  • All state behaviour — every kitty_states/ file is untouched

  • NavigationTarget and its adapters
  • InputHandler / InputSource / GodotInputSource

  • All existing Kitty behaviour at runtime — this is a rename/extract, not a redesign

  • No cast failed or Invalid type errors in the output log

Suggested Next Steps

Delete scripts/KittyDirector.gd from the filesystem

The file is now a stub comment. Godot will warn about the orphaned .uid file on next open — deleting both KittyDirector.gd and KittyDirector.gd.uid from the filesystem (Finder or terminal) cleans this up. Do this through the Godot editor's FileSystem panel to avoid stale uid references.

Rename scripts/kitty_states/ → scripts/states/

The states in this folder now belong to all characters, not just kitties. Renaming clarifies this. Do through the Godot editor (right-click → Rename) so all .uid references update automatically.

Execute the Separation Plan — update references first

The separation task plan was written before this refactor. Before executing, verify any KittyDirector references in that plan map to WorldDirector, and that the separation force property goes on Character (it already does, as that was added during this refactor).

Prove the base with a second character type

Create a minimal second character — even a re-skinned kitty with a different name and personality — to verify the Character base works end-to-end independently of Kitty. This is the acceptance test for the whole refactor. A new scene, extends Character, a CharacterPersonality resource, and confirm it spawns, animates, and takes AI decisions.

Introduce a NavigationState intermediate class

The base State class currently carries rotation_frames, _rotation_frame_count, and _tick_rotation() — machinery only used by StandUpState, SitUpState, and MovementState. A NavigationState extends State intermediate class concentrates this in the right place. Identified in the architecture review; now cleaner to implement since entity is typed as Character.

Design the Interactable Items system

The next major architectural milestone. Furniture, toys, pet houses — items with interaction spots, occupancy tracking, and capability-based auto-discovery. This warrants its own architecture session (grill/RFC) before any code is written. Key questions: how does a character discover nearby interactables, how does reservation prevent double-use, how do item-specific states clean up on removal.
