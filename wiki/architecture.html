1   The State Machine

Every kitty owns one FiniteStateMachine node. Each physics frame, the FSM calls state.tick(delta) on the active State. A state returns null to stay, or a new State instance to request a transition. The FSM does all transitions — states never call change_state() themselves.

The tick loop

  • _physics_process(delta) fires every physics frame
  • Calls state.tick(delta) on the active state
  • null return → stay in current state
  • New State instance return → call change_state()

Transition rules

  • Same-state guard: if the new state's name() equals the current name, transition is blocked (no re-entry)
  • Exiting state gets _exit_state() then queue_free()
  • New state: _enter_state(from), added to scene tree, set_physics_process(false)
  • Only the FSM drives _physics_process — states have it disabled

State ownership

  • States are Node children of the FSM
  • Each state gets a reference to its entity (Kitty) on enter
  • States access entity.anim, entity.navigation_target, etc.
  • Transition states chain: constructor receives the next state as a parameter

External triggers

  • Kitty.change_state(state) — direct override (buttons, AI SIT/LAY)
  • Kitty.begin_walk/run/sprint() — smart routing through stand-up / sit-up chains
  • Character.begin_manual_movement(input_handler, ...) — player W/A/S/D path; routes through rest/surface departure chains, then drives direct velocity
  • Character.apply_decision(decision) — entry point for AI brain outputs
The chaining pattern. Transition states like StandUpState and SitUpState receive their next state in their constructor. When the animation finishes, tick() returns that next state. This lets you express multi-step sequences as nested constructors: SitUpState.new(StandUpState.new(MovementState.walk())).

Common confusion points. "Stand Up" button plays stand-up then sit-down — this is correct, it previews the transition animation. After "Look Around" completes, the kitty freezes in sitting pose — also correct, it's waiting for the next AI or player decision. From LOOK_AROUND, the "Sit" button is not available because the kitty is already sitting. Use Walk/Run/Sprint to move, or Stand Up to preview the animation.

5   Autonomous AI (ActivityBrain)

Each autonomous character (not the player-controlled one) has an ActivityBrain. WorldDirector calls brain.tick(delta, viewport_rect) every physics frame. The brain decides when to fire a new activity and what it should be.

Decision cycle

  • _activity_timer accumulates delta every frame
  • When _activity_timer >= _duration[current_activity], a new decision fires
  • Timer resets to 0 after each decision
  • Decision is weighted-random; floor_type bias modifiers applied before sampling (WOOD boosts EXPLORE, STONE suppresses rest, RUG boosts sit/lay)
  • Result: an ActivityDecision (activity + optional walk target)
  • Passed to Character.apply_decision()

Fatigue system

  • _rest_timer accumulates the duration of each completed activity
  • When _rest_timer >= rest_threshold, the brain forces a LAY decision
  • Forces rest regardless of activity weights
  • _rest_timer resets to 0 after the forced rest
  • Default threshold: 10.0 seconds of accumulated activity

Walk target selection

  • Movement decisions (WALK, RUN, SPRINT) may include a walk_target
  • walk_target_chance probability of picking a random position within viewport bounds
  • If no target generated, kitty moves but may not have a destination

CharacterPersonality fields

  • Activity weights: preference_walk/sit/lay/run/look_around/sprint/explore — relative probability. 0 = never chosen.
  • Activity durations: how long each activity runs before next decision (seconds)
  • rest_threshold: total activity time before forcing lay-down rest
  • run_speed_multiplier (1.6) / sprint_speed_multiplier (2.4)
  • walk_target_chance: probability a movement decision includes a random nav-bounds position as its target

KittyPersonality fields (extends CharacterPersonality)

  • Look Around params: direction (BOTH/LEFT_ONLY/RIGHT_ONLY), speed, pauses, repetitions
  • These are accessed via as KittyPersonality cast in LookAroundState — returns null for non-kitty characters, which fall back to defaults
brain.tick(delta, random_walk_point, floor_type) → timer fires → _decide() → ActivityDecision → Character.apply_decision() → begin_activity() / begin_walk() / change_state(...)
EXPLORE activity: brain emits EXPLORE → apply_decision() calls _find_nearby_door() → if a Door is found, sets it as the walk target → begin_walk(). The character physically crosses into an adjacent room. EXPLORE probability scales with preference_explore (from KittyPersonality) and is boosted by the WOOD floor-type bias.

7   How to Describe a Bug

A good bug report gives enough context to reproduce the problem in one attempt. The debug console (backtick `) shows current state and a timestamped transition log — use it.

  1. What state was the kitty in? Use the exact state name from the State display in the debug console (e.g. LOOK_AROUND, not "looking around").
  2. What triggered the transition? Was it an AI decision, a button press in the debug console, or player input? If AI: how long had the current state been running?
  3. What animation played? Describe what you saw. Which direction was the kitty facing?
  4. What did you expect? Use the exact state names and behaviors from this document to describe the intended outcome.
  5. Which states were involved? List them in order: the state you were in, any intermediate states, the final state. Copy from the transition log if available.
  6. Is it reproducible? Can you reproduce it with a specific button press, or does it only happen with the AI? Reliable reproduction steps are worth more than a detailed description.

Example of a good bug report

"Kitty was in LOOK_AROUND (AI-triggered, had been running for ~2s). AI then decided SIT. Expected: kitty stays in sitting pose (SitState(true,true) — paused, held pose). Actual: kitty stood up slightly then sat back down — the full sit animation played from standing. Reproduced consistently by pressing Look Around, then immediately pressing Sit Up before the animation ends."

Anti-patterns

Too vague: "The kitty does the wrong animation sometimes." — no state, no trigger, not reproducible.

Missing trigger: "After looking around the kitty stood up." — was it an AI decision, a button press? From which exact state?

Wrong state names: "It was in the idle state" — use the enum names from this document (SIT, LAY, etc.).
