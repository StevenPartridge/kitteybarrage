Y-Sort Depth Rendering

Done

Sort kitties (and future characters) by screen Y so lower = in front. One scene node, minimal code.

Concepts & Vocabulary

Game Dev Aside — What You're Describing

Y-sorting (the common practical term) — sorting sprites by their Y screen position so that a character lower on screen draws on top of one higher up. It's the universal technique for top-down and isometric 2D games.

Painter's algorithm — the underlying principle: draw things from back to front. Things drawn later appear in front. Y-sorting is just an automatic, continuous application of this.

Top-down oblique projection — the implied camera. The world is viewed from above at an angle, so "down the screen" means "toward the viewer." Objects near the bottom of the screen are closer.

Z-index / draw order — in 2D engines there's no real 3D Z axis; instead each sprite has an integer z_index that determines what draws on top of what. Higher z_index = drawn later = appears in front. Y-sorting works by dynamically assigning z_index based on Y position each frame.

Godot's y_sort_enabled — Godot has this built in as a property on Node2D containers. When true, all direct children are sorted by their Y position automatically, every frame, with zero code.
Why the scene hierarchy matters

Godot's y_sort_enabled sorts the direct children of the node it's set on. For it to work, all characters that need to sort against each other must share the same immediate parent.

Right now, spawned kitties are added as children of KittyDirector (a Node, not a Node2D) — so y_sort_enabled can't be used on it directly. The fix is a one-node scene change.

Approach Options

OptionHowScene changesCode changesFuture-proof
A — YSortLayer node Add a Node2D with y_sort_enabled = true to the scene. Kitties go there. Add one node to Main.tscn 1 line in SpawnKittyAtLocation All future characters just go under the same node
B — Manual z_index Each frame, set kitty.z_index = int(kitty.position.y) in KittyDirector._physics_process. None ~3 lines added to existing loop Works, but other character types need to be included manually

Recommendation: Option A. It's the idiomatic Godot solution, zero ongoing CPU cost, and future characters automatically sort correctly just by being added to the same parent — no code changes needed when new character types are introduced.

Note: Option B has a subtle issue — z_index values are integers and are global across the scene. Kitties at Y=400 would get z_index = 400, which could conflict with other scene elements. y_sort_enabled handles sorting within its subtree cleanly without touching global z_index.

Scene Structure

Current hierarchy

Main (Node2D)
  KittyDirector (Node)        ← spawned kitties land here
  CanvasLayer                 ← DebugOverlay, UI

After this change

Main (Node2D)
  KittyDirector (Node)        ← logic only, no kitty children
  WorldLayer (Node2D)           ← NEW — y_sort_enabled = true
    Kitty, Kitty, Kitty...    ← all characters live here
  CanvasLayer                 ← unchanged

WorldLayer is a plain Node2D with one property set in the editor: y_sort_enabled = true. No script needed.

Scene pivot note: WorldLayer should sit at position (0, 0) with no scale or rotation — it's a transparent sorting container, not a transform. Give it a clear name so it's obvious what it does.

Implementation Steps

Add WorldLayer to Main.tscn

In the Godot editor, add a child Node2D to Main. Name it WorldLayer. In the Inspector, enable Y Sort Enabled. Position it between KittyDirector and CanvasLayer in the tree.

No script. No other properties to set.

Export a reference to WorldLayer in KittyDirector

Add one export to KittyDirector.gd:

@export var world_layer: Node2D

Wire it up in the editor: select KittyDirector, drag WorldLayer into the world_layer slot in the Inspector.

Add a null guard in _ready() so it fails loudly if unset:

func _ready() -> void:
    assert(world_layer != null, "KittyDirector: world_layer must be assigned in the editor")
    # ... rest of _ready
Spawn kitties into WorldLayer

In KittyDirector.gd, change one line in SpawnKittyAtLocation:

# Before
add_child(new_kitty)

# After
world_layer.add_child(new_kitty)

That's the only code change. The kitties array holds node references and doesn't care where in the tree they live — initialize_kitty, brain ticking, input handling, all work unchanged.

Verify auto-collection still works

Check KittyDirector._ready()'s sibling scan — it looks for Kitty nodes that are siblings of KittyDirector (children of Main). Since there are currently no pre-placed kitties in Main.tscn, this path is never hit, so nothing breaks.

If pre-placed kitties are ever added to the scene, they should go under WorldLayer, not directly under Main. The auto-collection loop would need to be updated at that point — but that's a future concern.

Future Characters

Any new character type (dogs, birds, toys, etc.) gets depth sorting for free by being added to WorldLayer. No code changes to the sorting logic ever needed — Godot handles it.

If a future character type is not managed by KittyDirector, its own director/spawner just needs a reference to WorldLayer and calls world_layer.add_child(new_character).

Origin point matters. Godot's y_sort_enabled sorts by each node's local Y position within the parent. For a CharacterBody2D like Kitty, the position is at the node's origin — which is ideally the character's feet, not their center. If the sprite is offset vertically in the scene, the sorting will still feel natural as long as the origin is consistently at the feet for all characters.

Testing Checklist

  • Spawn 3+ kitties — lower kitties visually appear in front of higher kitties
  • Move a kitty (walk/run) so it passes behind then in front of another — z-order updates smoothly
  • Spawn a kitty at the very top and very bottom of the screen — ordering is correct at extremes
  • Two kitties at nearly the same Y — no visible z-fighting or flickering
  • DebugOverlay (CanvasLayer) always renders above everything — unchanged
  • Right-click spawning still works — new kitties appear and sort correctly
  • Focus cycling (Tab) still works across all spawned kitties
  • Highlight on controlled kitty still visible when it's behind another kitty
  • No errors in the Godot output log related to WorldLayer or null references

Affected Files

  • scenes/Main.tscn — add WorldLayer node, wire export
  • scripts/KittyDirector.gd — add @export var world_layer, change add_child call
